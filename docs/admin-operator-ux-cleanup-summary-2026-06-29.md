# Npcink Workflow Toolbox 后台运营体验整理总结

日期：2026-06-29

## 背景

这一轮整理从后台设置页体验开始。最初的问题是：页面信息太杂、英文和中文混杂、工具名偏技术、入口过多，普通站长和运营人员很难判断“我现在应该点哪里、会不会改坏网站、下一步要做什么”。

本轮目标不是增加更多能力，而是重新整理后台产品面：

- 让入口按运营任务组织，而不是按内部 Ability、Core、Adapter、Cloud 等技术概念组织；
- 单张内容处理放到它实际发生的位置，例如文章编辑侧栏或媒体库图片详情；
- 批量处理才保留在 Toolbox 后台的图片处理面板；
- 低频、诊断、payload、原始输出、交接细节默认折叠；
- 所有最终 WordPress 写入继续走 Adapter、Core、Abilities 治理链路，Toolbox 不直接写媒体、文章、SEO 或设置。

## 主要使用者判断

当前插件后台主要给普通站长和运营人员使用，不是给开发者或治理系统维护者使用。用户关心的是：

- 站点哪里需要检查；
- 图片 ALT、图片体积、特色图、SEO 等是否有问题；
- 哪些建议可以放心提交；
- 哪些操作已经完成，哪些还需要人工审核；
- 出错后应该怎么恢复。

主界面不应直接暴露大量内部词，例如 raw payload、Ability ID、proposal handoff、Core-ready、workflow runtime、contract version 等。必要时这些信息可以保留，但应放进“高级”折叠区。

## 统一产品原则

1. 入口跟随任务场景。
   单篇文章相关能力放文章编辑侧栏；单张图片相关能力放媒体库图片详情；批量媒体任务放媒体库批量操作或 Toolbox 的批量图片面板。

2. 主按钮表达用户结果。
   按钮文案应是“扫描并建议 ALT”“提交并更新选中的 ALT”“提交优化预览”，而不是“构建 Core 交接草稿”“生成 proposal payload”。

3. 高级细节默认折叠。
   审计、payload、contract、raw output、blocked details 可以保留，但默认不打断运营路径。

4. 建议和写入必须分层。
   Toolbox 可以生成建议、构造提案、调用 Adapter 提交请求，但不直接执行最终 WordPress 写入。审批、执行、审计仍属于 Core、Adapter 和 WordPress Abilities。

5. 中文界面必须完整。
   运营界面不应出现残留英文。新增按钮、状态、错误提示、结果摘要都需要同步到 zh_CN 翻译文件。

## 已处理的后台入口问题

### 顶层结构

后台顶层 Tab 从技术分类调整为更贴近运营任务的结构：

- 概览；
- 站点画像；
- 图片处理；
- 内容准备；
- 高级。

“图片优化”不再作为顶层名称，因为它容易让用户误以为这里只处理压缩或单图优化。当前更准确的定位是“图片处理”：包含图片优化、批量 ALT、图片相关治理任务。

### 站点画像

站点画像页面从技术配置表单，调整为只展示稳定建议所必需的站点简介字段。SEO、AEO、GEO、声明边界等高级指导规则默认折叠。

原则是：普通运营人员先填写“站点定位、目标受众、品牌语气、关键词”等能理解的内容，高级规则需要时再打开。

### 内容准备

内容准备里曾经存在多个定位不清的入口：

- 文章准备包；
- 文章助手；
- 已审阅草稿交接；
- 内容机会检查；
- 图片候选采纳等。

处理后的原则是：

- 当前项目不生成整篇文章，因此 Article Assistant 这类“像写文章工作台”的后台入口没有价值，应移除产品入口；
- 没有真实外部稿件导入场景时，“已审阅草稿交接”不应作为后台常驻入口，只保留 REST 或 Ability 能力，未来给批量导入或 Cloud bulk import 使用；
- 内容机会检查如果缺少清晰运营闭环，应避免作为独立后台工具暴露；
- 单篇文章上下文能力优先放到文章编辑侧栏，而不是 Toolbox 后台深链接。

### 图片相关入口

单张图片处理不适合放在 Toolbox 后台工具页。更合适的位置是：

- 媒体库图片详情右侧；
- 媒体库编辑图片按钮附近；
- 文章编辑侧栏中当前文章使用图片的上下文面板。

Toolbox 后台保留批量或跨媒体库的任务，例如：

- 批量优化图片；
- 批量检查和补全图片 ALT；
- 查看被排除项目和失败原因；
- 提交已审阅的批量结果给 Core/Adapter。

## 已移除或降级的重复能力

本轮讨论中多次发现“单张能力放后台、批量能力没有批量选择、文章侧栏已有能力但后台又重复出现”的问题。处理原则如下：

- `ai-media-alt-suggestions`：不再作为旧的独立后台工具入口保留。当前批量 ALT 路径改为图片处理下的批量任务。
- `media-brief`：如果只是给指定文章推荐特色图，和文章编辑侧栏的 AI 特色图推荐重复，应移除后台入口。
- `optimize` 单张图片优化：不应继续作为 Toolbox 后台工具页入口，应迁移到媒体库图片详情或编辑按钮附近。
- `image-candidate-adoption`：如果是单张文章配图采纳，应放文章编辑侧栏；如果是批量采纳，则需要真实批量选择和结果处理能力，否则不应作为后台深链接保留。
- `article-assistant`：当前项目不生成文章正文，后台入口移除；无价值的 `npcink-toolbox/build-article-assistant` 能力也应清理。
- `article-plan`：短期没有外部稿件导入场景，后台入口移除，只保留 REST/Ability 能力给未来批量导入或 Cloud bulk import。

这些调整的核心不是删功能，而是减少普通运营人员看到的“半成品工作台”和内部流程入口。

## 批量图片优化的定位

批量图片优化的定位是：帮助运营人员找出可优化的媒体库图片，生成 Cloud 预览，人工确认后，通过 Adapter/Core 提交治理执行。

它解决的问题是：

- 媒体库图片体积过大；
- 单张优化入口效率太低；
- 需要在更新媒体文件前保留预览、审核和恢复证据。

它不应该变成：

- Toolbox 自己的批量写入器；
- 本地队列或调度器；
- 绕过 Core 的媒体替换工具；
- Cloud 或 Toolbox 的第二套审批系统。

当前更合理的体验是：

1. 运营人员选择范围和目标；
2. Toolbox 生成有限批量候选；
3. 用户预览并选择要提交的项目；
4. Toolbox 通过 Adapter 创建 Core 提案并请求执行；
5. Core 策略允许的项目自动通过，策略拦截的项目保留为审核状态；
6. 执行、审计、恢复证据由 Core/Adapter/Abilities 记录。

## 批量图片 ALT 的定位

批量图片 ALT 不是“生成一堆说明文字让用户复制”，而是一个受治理的批量补全流程：

- 扫描媒体库或指定范围内 ALT 缺失、较弱、像文件名、过长或关键词堆砌的图片；
- 给出建议 ALT；
- 运营人员逐行看图、修改、勾选；
- 提交选中行；
- Toolbox 通过 Adapter 创建 Core 提案并请求执行；
- Core 对满足条件的 ALT-only 项目自动通过，其他项目保留审核。

### 为什么可以类似批量优化做自动通过

因为“补全缺失或明显较弱的 ALT”在风险上低于替换图片文件、发布文章、修改 SEO 或批量改分类标签。但它仍然是媒体元数据写入，所以不能由 Toolbox 直接写。

合适边界是：

- Toolbox 负责扫描、展示、选择、提交；
- Adapter 负责连接 Core 和 Abilities；
- Core 负责审批策略、审计和是否允许自动通过；
- WordPress Abilities 负责最终执行。

### ALT-only 自动通过条件

Core 的 `smart_guarded` 策略只应自动通过非常窄的候选：

- ability 必须是 `npcink-abilities-toolkit/update-media-details`；
- 只能包含一个 `attachment_id`；
- 只能更新 `alt`；
- 不能包含 title、caption、description、source、attribution 等其他媒体字段；
- 输入保持 `dry_run=true`、`commit=false`；
- 来源必须是 Toolbox 的 `media_alt_caption_review_set.v1` 审阅行；
- 必须标记操作者已审阅并确认图片；
- 当前 ALT 状态必须是缺失、空、较弱、像文件名、过长或关键词堆砌；
- 调用方必须具备可信 Adapter 的 create/approve 权限；
- 自动通过仍受 Core 的小时和每日配额、审计写入和 preflight 约束。

如果混入 caption 或其他字段，不能走自动通过。

## 关于“为什么还要 Core 审核”

即使当前界面只有有权限的人能操作，也不应让 Toolbox 绕过 Core。

原因是：

- WordPress 后台权限只说明“谁能点按钮”，不等于说明“这次 AI 建议是否安全”；
- Core 负责记录提案、审批、preflight、执行结果和审计；
- 自动通过不是取消审核，而是把低风险、结构明确、证据充分的审核规则固化到 Core 策略里；
- 以后接入 OpenClaw、Cloud bulk import 或其他 Adapter 时，可以复用同一套治理结果。

因此正确做法是：允许 Core 策略自动通过窄范围项目，而不是让 Toolbox 直接写。

## 本地和云端的职责边界

批量图片和批量 ALT 的产品体验可以在本地 Toolbox 中呈现，但不应把本地插件变成运行时平台。

更合适的分工：

- Toolbox：固定按钮、表单、候选展示、人工选择、提交 Adapter；
- Cloud：必要时提供运行时细节，例如图片上下文证据、复杂分析或托管模型能力；
- Core：治理事实、审批策略、审计、preflight；
- Adapter：执行通道；
- Abilities Toolkit：可复用的 WordPress 能力定义。

不应在 Toolbox 中新增：

- 队列；
- 调度器；
- 租约；
- 重试工作器；
- 第二套 approval store；
- 第二套 workflow registry；
- 直接媒体写入器。

## 中文和提示文本问题

本轮反复出现的可用性问题包括：

- 部分按钮和状态仍是英文；
- 工具名带有内部术语；
- “Core 审核”“proposal”“handoff”等词直接暴露给运营人员；
- 状态提示没有解释“有没有真的改 WordPress”；
- 高级 payload 默认展开；
- 结果区把机器输出、检查清单、候选行、执行状态混在一起。

后续所有后台 UI 都应遵守：

- 主标题说用户任务；
- 副标题说会不会改内容；
- 按钮说结果动作；
- 成功提示说已完成什么；
- 阻塞提示说下一步去哪处理；
- 高级细节折叠；
- 新增 `t()` 字符串必须同步 zh_CN JSON、po、mo。

## 当前阶段结论

这轮工作的核心结论是：Toolbox 后台不应该继续暴露一堆“内部能力入口”，而应该成为普通运营人员可理解的固定任务面板。

已经形成的方向：

- 概览负责下一步任务；
- 站点画像负责稳定建议上下文；
- 图片处理负责媒体库批量任务；
- 内容准备只保留真实有用、定位清晰的编辑辅助；
- 高级区域只放低频诊断和治理细节；
- 单篇/单图能力移回文章编辑侧栏或媒体库详情；
- 批量写入类能力统一通过 Adapter/Core/Abilities，而不是 Toolbox 直写。

## 后续建议

1. 给媒体库图片详情补单图 ALT 和单图优化入口。
   后台不再保留单图 optimize 深链接，媒体库中才是用户自然入口。

2. 给媒体库列表补批量入口。
   例如批量操作中增加“扫描 ALT 建议”“优化所选图片”，让用户先选择具体图片，再进入 Toolbox 批量结果页。

3. 做一次全后台英文残留巡检。
   尤其检查 JS 动态文案、错误文案、按钮状态、折叠区标题。

4. 做一次旧 URL 清理。
   对已废弃深链接给出明确 fallback 或 redirect，避免用户进入空壳工具页。

5. 做一次真实运营流程走查。
   重点验证普通站长是否能回答三个问题：这页是干什么的、会不会自动改站点、下一步点哪里。

6. 在 Core 中继续保持窄策略。
   新增自动通过只能按候选类型逐个加入，不能引入通用“有后台权限就自动通过所有写入”的规则。
