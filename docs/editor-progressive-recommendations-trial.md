# Editor Progressive Recommendations Trial

Status: active operator trial template for the local progressive recommendation
slice.

This document records real-editor use of the local
`progressive_recommendations` layer. It is a product validation log, not a
request to add Cloud ranking, Core proposal creation, apply buttons, feedback
storage, or a workflow runtime.

## Trial Scope

Run the trial on 1-2 real human-written posts. Use posts with enough title,
excerpt, body text, taxonomy, or media context to judge whether local
recommendations are useful.

Focus only on the local progressive layer:

- local taxonomy/category/tag candidates;
- recent media-library candidates;
- local publish preflight warnings;
- candidate source labels and explanation quality;
- sidebar timing, refresh behavior, and review affordances.

Do not evaluate slower Cloud, image-generation, Site Knowledge deep search, or
Core proposal execution in this trial except to confirm they remain explicit
operator actions.

## Pre-Trial Gates

Run the repeatable gates before recording operator outcomes:

```bash
composer test:all
composer smoke:editor-progressive-recommendations
```

For local browser review, open the WordPress editor, open the Npcink Content
Support sidebar, confirm the automatic local progressive request completes
without showing a default success card, then open `Local suggestions` and use
View suggestions and Refresh without triggering a focused Cloud-backed tool.

## Outcome Vocabulary

Use the same decision labels for every candidate:

- `accepted`: the candidate is directly useful as-is.
- `edited`: the candidate is useful but needs human adjustment.
- `rejected`: the candidate is wrong, noisy, or irrelevant.
- `missing`: an expected local candidate was absent.
- `blocked`: the local post/site context was insufficient to judge.

Use the same action class labels for every candidate:

- `informational`: review-only guidance such as preflight or SEO reminders.
- `copyable`: useful text or term guidance that an editor can manually reuse.
- `handoffable`: a write-like target that must stay on a Core or explicit
  future local-consent path.

## Triage Rules

Fix only narrow regressions found during this trial:

- automatic prefetch calls Cloud;
- automatic prefetch creates or submits a Core proposal;
- local recommendation review mutates WordPress content;
- stale or late REST responses replace newer fingerprint results;
- timeout or sidebar unmount creates visible editor instability;
- candidate copy hides source evidence or implies automatic application.

Do not widen scope during this trial:

- no Cloud rerank;
- no automatic proposal creation;
- no apply buttons for progressive recommendations;
- no persistent feedback or learning store;
- no workflow run, retry, queue, scheduler, or approval status;
- no new write path inside Toolbox.

## Exit Criteria

The progressive layer is ready for wider editorial use when:

- the panel loads reliably in the editor within the 2.5 second UX budget;
- every automatic result is local-only and no-write;
- candidate sources and action classes are clear enough for operator review;
- taxonomy, media, and preflight recommendations have enough accepted or edited
  outcomes to justify further local ranking work;
- rejected outcomes identify tuning changes rather than forcing new product
  scope;
- article text, metadata, media, terms, and publish status remain unchanged
  unless a separate explicit Core or local-consent path is used.

If these criteria fail, fix the smallest local-ranking, UI-copy, or lifecycle
issue and rerun only the affected trial step. Do not add a large AI
"all recommendations" request to compensate for local signal gaps.
